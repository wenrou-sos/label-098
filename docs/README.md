## 📦 安装步骤

### 方式一：一键启动（推荐）

项目包含完整的启动脚本，分别在两个终端运行：

**步骤 1: 启动后端服务**
```powershell
# 终端1 - 进入后端目录
cd d:\proje\label-098\backend

# 安装Python依赖（首次运行）
pip install -r requirements.txt

# 启动后端服务
python app.py
```
✅ 后端启动成功会显示：
```
🌐 服务地址: http://localhost:5000
 * Running on http://127.0.0.1:5000
```

**步骤 2: 启动前端服务**
```powershell
# 终端2 - 进入前端目录
cd d:\proje\label-098\frontend

# 安装Node依赖（首次运行）
npm install

# 启动前端开发服务器
npm run dev
```
✅ 前端启动成功会显示：
```
  VITE v5.x.x  ready in xxx ms
  ➜  Local:   http://localhost:3000/
```

**步骤 3: 访问系统**
打开浏览器访问：**http://localhost:3000** 🎉

- 按5个年龄段分别统计男女用户对4项条件的硬性要求比例

**图表说明**：
1. **雷达图（左上）**：越靠外表示该维度要求越普遍
   - 女性在收入/房产维度通常高于男性

2. **趋势折线图（右上）**：
   - 实线 = 学历要求，虚线 = 收入要求

3. **分组柱状图（下方）**：

   - 可直观比较同性别不同年龄的要求变化

**典型发现**：
- 女性择偶要求通常在31-35岁达到峰值
- 男性对身高的要求随年龄增长略有下降
- 学历要求在双方都呈"倒U型"曲线

### 模块四：😰 婚恋焦虑指数

**功能说明**：
- 构建综合焦虑指数模型（付费率+客单价+年龄+房产+收入）
- 仪表盘直观展示各线城市焦虑等级
- 散点图分析城市付费意愿与客单价关系
- VIP等级分布与城市付费榜

**焦虑指数构成**：
```
极度焦虑 (红色)   ≥ 75分
高度焦虑 (橙色)  55-74分
中度焦虑 (蓝色)  35-54分
轻度焦虑 (绿色)  < 35分
```

**影响因素权重**：
1. **付费意愿 (35%)**：是否愿意为婚恋服务买单 → 直接反映焦虑程度
2. **平均客单价**：消费金额越高 → 焦虑越强
3. **30岁以上占比 (25%)**：大龄未婚比例 → 社会压力传导
4. **无房占比 (20%)**：物质条件不足 → 婚恋门槛焦虑
5. **低收入占比 (15%)**：经济基础薄弱 → 择偶竞争力不足

**散点图阅读技巧**：
- **X轴**：付费率（越高越愿意花钱）
- **Y轴**：平均客单价（越高单次消费越多）
- **气泡大小**：用户规模（越大代表用户越多）
- **气泡颜色**：城市线级（红=一线，橙=新一线，蓝=二线，绿=三线）

### 模块五：💕 脱单成功规律

**功能说明**：
- 深入分析4000+成功牵手案例
- 年龄差/学历差/收入差三维度分布规律
- 5×5学历/收入组合热力图
- 核心规律指标汇总

**核心规律指标解读**：

| 指标 | 典型值 | 解读 |
|------|--------|------|
| 男方年龄偏大占比 | ~65-75% | "男大女小"仍是主流模式 |
| 年龄差3岁内 | ~55-65% | 年龄越相近越容易成功 |
| 学历相同占比 | ~40-50% | "门当户对"现象显著 |
| 男方学历更高 | ~30-35% | 男方学历略高于女方也常见 |
| 收入相同占比 | ~30-40% | 经济条件相近利于稳定 |
| 男方收入更高 | ~45-55% | 传统"男强女弱"收入模式占优 |

**热力图阅读指南**：
- **对角线** = 条件完全相同（门当户对）
- **左下角 → 右上角** = 男方条件越来越好
- **颜色越深** = 该组合成功案例越多
- **观察热点区域** = 成功率最高的条件组合

## 🛠️ 生产环境部署

### 后端部署（Gunicorn + Nginx）

```bash
# 1. 安装Gunicorn
pip install gunicorn

# 2. 生产模式启动
cd backend
gunicorn -w 4 -b 0.0.0.0:5000 app:app

# 3. Nginx配置示例（反向代理）
# /etc/nginx/conf.d/dating-dashboard.conf
server {
    listen 80;
    server_name your-domain.com;
    
    location /api/ {
        proxy_pass http://127.0.0.1:5000/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

### 前端部署（静态构建）

```bash
# 1. 构建生产版本
cd frontend
npm run build

# 2. 构建产物在 dist/ 目录
# 上传 dist/* 到任意静态服务器（Nginx/Apache/OSS/CDN）

# 3. Nginx配置示例
server {
    listen 80;
    server_name your-domain.com;
    root /var/www/dating-dashboard/dist;
    
    location / {
        try_files $uri $uri/ /index.html;
    }
    
    location /api/ {
        proxy_pass http://backend-server:5000/api/;
    }
}
```

### Docker部署（可选）

可编写Dockerfile将前后端分别容器化，使用docker-compose编排。

## ❓ 常见问题 FAQ

### Q1: 启动后端报错 `ModuleNotFoundError: No module named 'xxx'`
**A**: 未安装Python依赖，执行：
```bash
cd backend
pip install -r requirements.txt
```

### Q2: 启动前端报错 `Cannot find module 'xxx'`
**A**: 未安装Node依赖，执行：
```bash
cd frontend
npm install
```

### Q3: 前端页面加载但无数据，控制台报网络错误
**A**: 检查后端服务是否启动正常
- 确认 `http://localhost:5000/api/health` 可访问
- 确认Vite代理配置正确（vite.config.js中端口5000对应后端）

### Q4: 数据生成太慢/想减少数据量
**A**: 修改 `backend/data_generator.py` 中的参数：
- 第271行 `n_users = 50000` → 改为更小值（如5000）
- 第315行 `count=15000` → 匹配记录数对应减小

### Q5: 图表中中文显示为方框/乱码
**A**: 这是ECharts字体问题，通常现代浏览器自动处理
- 建议使用 Chrome/Edge 最新版
- 如仍有问题，检查操作系统是否安装中文字体

### Q6: 如何接入真实业务数据？
**A**: 
1. 替换 `data_generator.py` 为真实数据库查询
2. 保持输出的4个CSV文件结构不变（或直接在内存中返回DataFrame）
3. 分析引擎 `analyzer.py` 无需任何修改即可适配真实数据
4. 如需增量更新，修改 `app.py` 中的缓存刷新策略

### Q7: 内存不足或数据量大时报错
**A**: 
- 减少数据量（参见Q4）
- 使用Dask替代Pandas处理百万级以上数据
- 增加服务器内存配置
